# Sprint 1 Execution Plan

## Goal
Перестроить admin-first MVP на разделенные сущности `клиент/велосипед/аренда` и получить рабочий поток:
`создать клиента -> создать велосипед -> создать аренду`.

## Frozen Decisions (Week 1)
- Формулы оплаты:
  - `1 день = неделя / 7`
  - `2 недели = неделя * 2`
  - `1 месяц = неделя * 4`
- Округление сумм: до десятков рублей вверх (пример: `431 -> 440`).
- Логика долга: долг считается по ledger, ручная корректировка хранится отдельными записями и суммируется.
- Регистрация клиентом не нужна: доступ клиенту выдается админом при создании аренды (`login/password`).
- MVP язык: только русский.
- Push в MVP: не требуется.
- Фокус реализации: сначала iOS admin часть, Android позже (после стабилизации UI/UX на iOS).

## Tasks
1. Backend
- Ввести модель `Bike` и отвязать client от bike/rate.
- Перевести `Rental` на ссылки `clientId + bikeId`.
- Реализовать API:
  - `GET/POST /api/v1/admin/clients`
  - `GET/POST /api/v1/admin/bikes`
  - `POST /api/v1/admin/rentals`
- В `POST /admin/rentals` добавить блок доступа клиента:
  - валидация login/password
  - создание или обновление учетных данных клиента.
- Обеспечить человекочитаемые ошибки без падения приложения.

2. Database
- Хранить и восстанавливать 3 сущности после рестарта backend.
- Обновить debug views в pgweb:
  - `clients_view`
  - `bikes_view`
  - `rentals_view`
  - `user_credentials_view`
- Проверить атомарность сохранения состояния.

3. Mobile KMM
- Обновить shared API-модели под новые сущности.
- iOS:
  - Сервисный экран с карточками `Новый клиент` и `Новый велосипед`.
  - Форма `Новая аренда` с выпадающими списками клиентов/велосипедов.
  - Загрузка фото велосипеда из галереи и отправка на backend.
  - Блоки `Доступ клиента` и `Документы и комментарии` в новой аренде.

4. DevOps
- Стабильный запуск backend + БД UI одним скриптом.
- Проверка статуса и health-check без ложных негативов.
- Логи и диагностика в читаемом виде для дебага.

## Week 1 Detailed Sequence
1. Day 1-2: Data model and contracts
- Выделить независимые сущности `Client`, `Bike`, `Rental`.
- Убрать bike/rate из `Client`.
- Перевести `Rental` на `clientId + bikeId`.
- Зафиксировать API контракты `clients/bikes/rentals`.

2. Day 2-3: Backend CRUD + validation
- Реализовать list/create клиентов и велосипедов.
- Реализовать создание аренды через выпадающий выбор `client` и `bike`.
- В аренде добавить блок `Доступ клиента`: создать/обновить client-user (`login/password`).
- Проверить человекочитаемые ошибки для всех обязательных полей.

3. Day 3-4: Persistence and resilience
- Сохранение всех сущностей в PostgreSQL после каждой мутации.
- Восстановление состояния после рестарта backend.
- Проверка сценариев внезапного перезапуска.

4. Day 4-5: iOS admin flows
- Вынести `Новый клиент` и `Новый велосипед` в сервисный экран.
- Сделать `Новая аренда` отдельной формой (основной рабочий сценарий).
- Добавить фото велосипеда из галереи с передачей на backend.
- Оставить в аренде блоки `Доступ клиента` + `Документы и комментарии`.

5. Day 5-6: Tests and hardening
- Unit-тесты валидации и ViewModel на create client/bike/rental.
- Backend integration tests на новые маршруты.
- UI-тесты критичного админ-пути создания клиента.
- Проверка отсутствия падений: все ошибки только в интерфейс.

6. Day 7: Stabilization
- Прогон smoke на simulator + real iPhone.
- Проверка читаемости ошибок и устойчивости к network hiccups.
- Закрытие оставшихся багов по admin-first сценарию.

## Done Criteria
- Создание клиента не требует полей аренды/велосипеда.
- Создание велосипеда работает отдельно.
- Создание аренды работает через выбор из существующих клиента и велосипеда.
- После перезапуска backend созданные клиенты/велосипеды/аренды сохраняются.
- iOS admin flow не падает и показывает понятные ошибки в интерфейсе.
