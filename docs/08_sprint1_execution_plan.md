# Sprint 1 Execution Plan

## Goal

Перестроить admin-first MVP на модель из `docs/14_rental_lifecycle.md`:

```text
создать велосипед -> создать Аренду для велосипеда -> создавать и закрывать клиентские аренды внутри этой Аренды
```

## Frozen Decisions

- `Аренда` - карточка жизненного цикла велосипеда на главном экране админа.
- `Клиентская аренда` - отдельный период клиента с собственными логином, паролем, ledger, платежами и датами.
- Один велосипед может быть привязан только к одной неудаленной `Аренде`.
- Одна `Аренда` может породить неограниченное число `клиентских аренд`.
- Статусы `Аренды`: `long_term`, `soon_return`, `mine`.
- Переход в `mine` закрывает текущую `клиентскую аренду`, но не удаляет карточку `Аренды`.
- Новый цикл из `mine` создает новую `клиентскую аренду` в существующей `Аренде`, а не новую `Аренду`.
- Логин и пароль принадлежат `клиентской аренде`.
- Старые логины закрытых `клиентских аренд` продолжают работать для просмотра и оплаты долга.
- MVP язык: только русский.
- Push в MVP: не требуется.
- Фокус реализации: сначала iOS admin часть, Android позже после стабилизации UI/UX на iOS.

## Tasks

1. Backend domain
- Ввести модель `ClientRental`.
- Перевести `Rental` на связь `bikeId` без обязательного `clientId`.
- Добавить `pipeline_status = long_term|soon_return|mine`.
- Обеспечить уникальность неудаленной `Rental` на `bikeId`.
- Перенести `login/password` из клиента/аренды в `ClientRental`.
- Добавить безопасную проверку неповторяемости паролей.
- Реализовать закрытие `ClientRental` сегодняшней датой.
- Реализовать старт нового `ClientRental` внутри существующей `Rental`.
- Реализовать удаление `Rental` с закрытием текущего `ClientRental` и переносом долга на клиента.

2. Backend API
- Обновить маршруты:
  - `GET/POST /api/v1/admin/bikes`;
  - `GET/POST /api/v1/admin/rentals`;
  - `GET /api/v1/admin/rentals/{rentalId}`;
  - `POST /api/v1/admin/rentals/{rentalId}/pipeline-status`;
  - `POST /api/v1/admin/rentals/{rentalId}/finish`;
  - `POST /api/v1/admin/rentals/{rentalId}/client-rentals`;
  - `DELETE /api/v1/admin/rentals/{rentalId}`;
  - `GET /api/v1/admin/clients?available_for_rental=true`.
- Все ошибки валидации должны быть человекочитаемыми.

3. Database
- Обновить таблицы по `docs/05_db_schema.sql`.
- Добавить миграцию старых данных:
  - текущие `rental` с клиентом превращаются в `rental + active client_rental`;
  - `pipeline_status` сохраняется;
  - ledger/payment записи получают `client_rental_id`.
- Обновить debug views:
  - `rentals_view`;
  - `client_rentals_view`;
  - `clients_view`;
  - `bikes_view`;
  - `client_rental_credentials_view`.

4. iOS admin
- Главный экран показывает список `Аренд`.
- Карточка аренды открывается на весь экран.
- В статусе `mine`:
  - статистика и журнал текущей аренды скрыты/заменены прочерками;
  - клиент отвязан;
  - выбор клиента открывает отдельный экран выбора;
  - логин и пароль редактируемые;
  - кнопка копирования валидирует заполненность;
  - `+ Корректировка` неактивна;
  - `Создать новую!` запускает новый `ClientRental`.
- Добавить нижнюю навигацию:
  - главный экран;
  - список клиентов;
  - список велосипедов.

5. Client app
- Вход клиента резолвит конкретный `client_rental`.
- Активная `client_rental` работает по текущим расчетам.
- Завершенная `client_rental` показывает дату завершения и фиксированный долг.
- Оплата завершенной `client_rental` уменьшает долг и не создает новых начислений.

6. Tests
- Backend:
  - нельзя создать две неудаленные `Rental` на один велосипед;
  - `finish` закрывает `ClientRental` и переводит `Rental` в `mine`;
  - новый цикл из `mine` не создает новую `Rental`;
  - завершенная `ClientRental` не получает новые начисления;
  - удаление `Rental` переносит долг на клиента.
- iOS:
  - выбор свободного клиента;
  - генерация и копирование доступа;
  - запуск нового цикла;
  - отсутствие дубликата аренды на главном экране.

## Done Criteria

- Создание клиента не требует полей аренды, велосипеда, логина или пароля.
- Создание велосипеда работает отдельно.
- Создание `Аренды` работает только для свободного велосипеда.
- `Аренда` в статусе `mine` может начать новый цикл без создания новой `Аренды`.
- Долги завершенных `клиентских аренд` сохраняются и оплачиваются.
- После перезапуска backend все `bike`, `rental`, `client_rental`, ledger и платежи сохраняются.
